# Design

This change generalizes the React-only vendor sharing landed in
`2026-06-26-share-published-vendor-runtime`. Its decisions D1–D5 still hold; the
notes below extend them. Where a decision below renumbers, it supersedes the prior
one for the generalized path.

## D1. One `SharedRuntime` descriptor is the single source of truth

The prior change kept three independently-hardcoded React lists that had to stay in
sync by hand: `VENDOR_MODULES` (publish.ts), `browserExternal` (publish.ts), and the
four importmap keys (documents.ts). Generalizing multiplies the drift hazard, so the
three collapse into one descriptor computed once per publish:

```
SharedRuntime = Array<{
  package: string         // 'react', '@emotion/react', '@acme/design-tokens'
  specifiers: string[]    // ['react', 'react/jsx-runtime'] — bare + declared subpaths
  origin: 'published' | 'workspace'   // governs SSR treatment (see D6)
}>
```

Everything else is derived from it, by construction, so the three can never disagree:

- **`external`** for every chrome / per-component / primer browser build = the flat
  list of all `specifiers`.
- **vendor entrypoints** = one generated entry file per specifier (D3).
- **importmap** = `specifier → that specifier's hashed vendor URL`, threaded through
  `BuildDescriptor.assets` / `DocAssets` as a `Record<string,string>` instead of the
  single `vendor` URL.

## D2. The vendor build is bounded, so `splitting` is safe *here* (and only here)

The reason `splitting` cannot help the catalog is its unbounded size — that is the
whole crash-containment precondition, and it is unchanged: the catalog stays one
isolated `Bun.build` per component. But the **shared set is small and bounded** (a
handful of packages), so the vendor build may do the one thing the catalog cannot:
build *all* shared specifiers' entries in a **single `Bun.build` with
`splitting: true`**.

- Splitting across that one build deduplicates cross-package internals automatically
  — the React reconciler imported by both `react-dom` and `react-dom/client` lands in
  one shared chunk instead of being inlined into each vendor entry. This is splitting
  doing exactly what it is good at, on a graph small enough that it is permitted to.
- Output: one hashed entry file per specifier + shared chunks. The importmap maps each
  specifier to its entry file; the shared chunk is fetched once.
- This *replaces* the prior single combined `vendor-react` bundle. It is the same
  shape of work — one bounded isolated worker build — so the `scalable-serving`
  isolation guarantee is untouched; it merely emits multiple entries with splitting.

This is the direct answer to "splitting can't reach across the per-component builds":
we never need it to. We only need it across the bounded vendor set.

## D3. Each vendor entry is generated by introspection (unchanged form, now per specifier)

The CJS-robust form from the prior D2 is retained and applied **per specifier**:
`import * as ns from '<spec>'` (a namespace import retains the module's whole impl —
nothing tree-shaken) + `export const x = ns.x` for every name the *installed* module
exports (a static binding the importmap can resolve at link time). It tracks the
consumer's installed version, works for CJS and ESM alike, and first-module-wins on
a duplicate name. One generated entry file per specifier; all are entrypoints of the
single D2 build.

Retaining each shared module *whole* (no tree-shaking) is the deliberate cost that
makes sharing robust — and it is what makes the selection policy in D5 conservative.

## D4. Sharing is the general form of `pinReact`

`pinReact` exists because two React instances break hooks. Marking a package
`external` + importmap-resolved means every surface's bare import resolves to the
**one** vendor instance — which subsumes pinning for that package, exactly as the
React change did. So declaring `share: ['@emotion/react']` is not only a size win: it
*guarantees* the single style cache / context a CSS-in-JS engine needs to function.
This is the single-instance-correctness property the spec's *Author-declared shared
libraries* requirement states. `pinReact` itself remains for the dev server and the
always-shared React default; declaring a package shared generalizes its effect to that
package on the published build.

## D5. Selection policy: safe default, explicit declaration, advisory discovery

Three tiers, in order of authority:

1. **Always-on (default): the React family.** Correctness + universal; output stays
   byte-identical to today when nothing else is declared.
2. **Author-declared: `share?: string[]` in config.** The author lists packages —
   their style-engine packages, `markdown-to-jsx`, shared utilities, and monorepo
   workspace package names. Declarative-first (not auto-hoist-everything) because the
   D3 form retains each module whole: for a large library where each component uses a
   thin slice, sharing-the-whole-thing-once only wins past a crossover (roughly when
   `copies × tree-shaken-slice > whole-module`). React is always past it; a
   rarely-shared library can *regress*. Declaration keeps it deterministic and
   intentional. Why a config field and not auto-detection of the style engine:
   `styleEngines` are factory **functions**, not package names (index.ts), so Display
   Case cannot name the engine's package from config — the author must.
3. **Advisory discovery (Phase 3): report, don't impose.** Each build already records
   its real module graph (`graphRecorder` → `inputs`). After a publish, report which
   `node_modules` / workspace packages are inlined across more than one component and
   how many bytes that duplicates, recommending `share` entries. This surfaces the
   size-only opportunities ("no silent caps") without taking on the regression risk of
   automatic hoisting.

### Subpath / deep-import scope

A declared package shares its **bare specifier plus any explicitly listed subpaths**
(`'@acme/icons/solid'`). A package's `exports` map MAY later be read to enumerate
subpaths, but v1 stays conservative: a deep import that is not a declared specifier
stays inlined — graceful, still correct (the spec's render-correct fallback), and the
Phase 3 report surfaces it. Trailing-slash importmap prefixes are out of scope (they
need the package's internal file tree served, which the one-bundled-file model does
not provide).

## D6. Monorepos: where browser and SSR must diverge

The client-download win is purely browser-side; SSR bundles run on the host and are
not downloaded. So the two sides are treated by `origin`:

- **Published shared package** (resolves to a registry version): share on the client
  (vendor bundle + importmap); keep `external` on the SSR renderers (as React is
  today); and **add it to the generated showcase `package.json` dependencies** with
  the consumer's installed version range, so the deployed `bun install` provides it at
  runtime. This generalizes the currently-hardcoded `react`/`react-dom` deps.
- **Repo-internal workspace package** (no registry coordinates): share on the client
  the same way (content-hashing handles its fast churn — change it, new hash,
  re-fetched; unchanged, cached: a *cache-granularity* win as well as a size win), but
  **bundle it into each SSR renderer** rather than externalize, because a private
  package cannot be `bun install`ed into the deployed showcase. Duplicating it across
  SSR renderers is host disk, not client bytes — not worth shipping private packages
  into the deploy. So the SSR `external` set stays "published shared packages only,"
  and the client sharing is what the spec's "published or defined within the same
  repository" clause guarantees.

## D6.1. Externalize the EXACT declared specifiers, not Bun's package prefix

A browser surface must externalize a shared package's *declared* specifiers but
**inline any undeclared subpath** — otherwise the spec's render-correct fallback
breaks. Bun's built-in `external: ['markdown-to-jsx']` matches by package *prefix*, so
it also externalizes an internal `markdown-to-jsx/entities` import. That subpath has no
importmap entry (only declared specifiers do), so the browser hits
`Failed to resolve module specifier "markdown-to-jsx/entities"` and the surface fails
to hydrate. (Found by the D8 headless load, not by types — exactly why D8 is a real
browser, not a type pass: every render page passed; only the primer, which bundles the
renderer via an absolute path through the mdx plugin and so carries the internal
subpath import, failed.)

Fix: a resolve plugin (`externalExact`) marks external only the *exact* declared
specifiers; everything else resolves and bundles normally, so an undeclared subpath
inlines. Browser builds use it; **SSR keeps Bun's prefix `external`**, because there a
bare subpath resolves from the deployed `node_modules` at runtime (prefix matching is
correct, and the package is in the generated `package.json`). Guarded by a
`publish.test.ts` regression test: the `consumer-shared` fixture's primer must inline
`markdown-to-jsx/entities` rather than emit it as a bare import.

## D7. Phasing keeps each step shippable and de-risks the review concern

- **Phase 1** lands the mechanism with the React-only default → byte-identical output,
  under the existing spec, immediately making the generic path *real* (answering "don't
  treat the generic case as solved") without changing any user's build.
- **Phase 2** adds `share`, the generalized `package.json` deps, and the
  published-vs-workspace SSR split — the user-facing feature; the new spec
  requirements land here.
- **Phase 3** adds the duplicate-runtime report.

Each phase is its own PR + changeset.

## D8. Verification is a real browser load (unchanged discipline)

Per the prior D5: a type pass cannot prove an importmap resolves. Each phase that
changes delivered bytes is verified by publishing this repo's showcase and
headless-loading the shell + several render pages, asserting each hydrates (`#root`
populated) with **zero** console/page errors, plus the static export containing the
importmap. Phase 2 adds a fixture with a declared non-React shared library (and, where
feasible, a workspace package) to prove the generalized path end-to-end — one shared
bundle for it, a per-component bundle importing it as an external bare specifier
(re-inlining guard), and a single-instance assertion for a context-bearing library.

## Risks

1. **Tree-shaking regression** for a low-frequency shared library (D3 retains modules
   whole) → conservative declarative default + the Phase 3 report; never auto-hoist.
2. **Subpath coverage** → scoped to bare + declared specifiers; documented; report
   surfaces unmapped duplicates.
3. **CJS/ESM introspection on a non-React library** → the namespace form handles both;
   proven by a non-React fixture (D8).
4. **`exports` conditions** (`"browser"`) → the vendor build is `target: 'browser'`, so
   Bun honors the browser condition; verified on the fixture.
5. **Version skew** (two majors of one library in a repo) → sharing assumes a single
   version (true for peer deps by definition); such a library should not be declared
   shared. Documented.
