# Design

## D1. Mechanism: external + importmap, not code-splitting

`splitting: true` would dedup shared code across entrypoints — but only within a
*single* `Bun.build`. The published catalog is built as one isolated `Bun.build`
*per component* (the crash-containment precondition: no pass ever holds the whole
catalog). So splitting can't reach across them. Instead: mark the runtime
`external` in the chrome and per-component browser builds (they emit bare
`import … from "react…"`), build the runtime **once** into a shared bundle, and emit
an `<script type="importmap">` in each document mapping the bare specifiers to that
one bundle's URL. Importmaps are static markup, so this works under a plain static
host too — no runtime resolution server needed.

## D2. Why the vendor entry is generated by introspection, not `export *`

React ships as CommonJS. The vendor bundle must re-export every name a surface
imports as a **static ESM binding**, because the importmap resolves a surface's
`import { useState } from "react"` against the vendor module's export list at link
time. Two naive forms both fail:

- `export * from "react"` over CJS yields runtime property *copies*, not static
  export bindings — `import { useState }` throws "Export 'useState' is not defined
  in module" in the browser.
- `export { createRoot } from "react-dom/client"` gives a static binding but lets
  the bundler tree-shake the reconciler it can't see `createRoot` reach through the
  CJS boundary — leaving a stub (the bundle shrank to ~13 KB, and the browser threw
  on the missing reconciler export).

The robust form, generated per build: `import * as ns from "<spec>"` (a namespace
import retains the module's **whole** implementation — nothing tree-shaken) plus
`export const x = ns.x` for every name the installed module actually exports (a
**static** binding). The names are read from the consumer's *installed* modules
(`Bun.resolveSync` + `import`), so the vendor tracks their exact version rather than
a hardcoded API list. First module wins a duplicate name (`Fragment` → `react`, not
`react/jsx-runtime`; `version` → `react`), which is correct because the duplicates
are the same value.

## D3. Which specifiers are external / mapped

`react`, `react-dom`, `react-dom/client`, `react/jsx-runtime` — everything a
browser surface imports (the SSR-only `react-dom/server` and the dev-only
`react/jsx-dev-runtime` never reach a browser bundle, the latter because publish
defines `NODE_ENV=production`). All four importmap entries point at the one vendor
bundle; each surface's import picks its names from the shared namespace.

## D4. The vendor build pins the consumer's React; externalized builds carry none

The vendor build keeps `pinReact: true`, so the single consumer React copy is what
lands in the shared bundle. The chrome and per-component builds set
`pinReact: false` + `external` — React isn't in them at all, so there's no dual-React
hazard to pin against; every bare import resolves through the importmap to the one
vendor copy. This *subsumes* pinReact's old job on those builds (collapse to one
consumer React) via a different route.

## D5. Verification is a real browser load

A type-checking pass can't prove the importmap actually resolves and React runs.
The change is verified by publishing this repo's showcase and loading the shell and
several component render pages headless, asserting each hydrates (`#root` populated)
with **zero** console/page errors — i.e. `createRoot`, hooks, and JSX all resolve
through the importmap. The static export is checked to contain the importmap too.
